コードレビューの定型チェックは、OpenClawのスケジュール実行を使うと継続的に自動化できます。最初から大きな範囲を任せるのではなく、対象を絞って運用ログを確認しながら広げるのが現実的です。

結論: OpenClawでコードレビューを自動化するワークフローは、最小構成で始めてログを見ながら段階的に広げるのが最短です。

## Background

コードレビューでは、差分の確認、テスト観点の洗い出し、命名や設計の指摘など、繰り返し発生する作業があります。これらをすべて手作業で行うと、レビュー品質が担当者やタイミングに左右されやすくなります。

OpenClawを使えば、決まった時間や条件でレビュー補助タスクを実行できます。まずは毎日の確認や特定ブランチのレビュー要約など、小さく始めるのが安全です。

## Step-by-step

1. 目的と対象範囲を決める

最初に、自動化するレビュー作業を1つに絞ります。たとえば「mainブランチとの差分を確認して、リスクと確認観点を要約する」など、成功条件を明確にします。

対象リポジトリ、対象ブランチ、確認するファイル範囲も先に決めておきます。範囲が広すぎると、実行時間が伸びたり、通知内容が曖昧になったりします。

2. 最小構成でスケジュールを設定する

まずは毎日1回、決まった時刻にレビュー補助ジョブを実行します。タイムゾーンは明示して、意図しない時刻に動かないようにします。

```bash
# 毎日9時に、隔離されたセッションでレビュー補助ジョブを実行する
openclaw cron add \
  --name "daily-code-review" \
  --cron "0 9 * * *" \
  --tz "Asia/Tokyo" \
  --session isolated \
  --message "mainブランチとの差分を確認し、コードレビューで見るべきリスク、テスト観点、注意点を日本語で要約してください。"
```

`--session isolated`を使うと、ジョブごとに独立したセッションとして実行できます。定期実行では、前回の作業状態に引きずられにくい構成にしておくと検証しやすくなります。

3. 実行結果を検証する

初回実行後は、ログ、実行時刻、出力内容を確認します。レビュー観点が具体的か、対象外のファイルに触れていないか、通知文が長すぎないかを見ます。

失敗時の再実行手順も決めておきます。認証切れ、依存サービス停止、対象ブランチ不在など、よくある原因ごとに確認コマンドや復旧手順を残しておくと運用が安定します。

4. 対象範囲を段階的に広げる

最小構成が安定したら、対象ブランチや実行タイミングを増やします。たとえば、毎朝のレビュー要約に加えて、リリース前だけ追加のチェックを走らせる構成にできます。

ただし、一度に複数の変更を入れると問題の原因が追いにくくなります。スケジュール、プロンプト、対象範囲は1つずつ変更して、ログで効果を確認します。

## Common pitfalls

- タイムゾーンを指定し忘れて、UTC基準でジョブが実行される  
  `--tz "Asia/Tokyo"`を明示し、初回実行後に実際の実行時刻をログで確認します。

- レビュー対象が広すぎて、出力が抽象的になる  
  最初は特定ブランチや特定ディレクトリに絞ります。必要に応じて、メッセージ内で「変更ファイルに限定する」「重大なリスクを優先する」と指定します。

- 認証切れでリポジトリや外部サービスにアクセスできない  
  定期実行に使う認証情報の有効期限を確認します。失敗時のログに認証エラーが出ていないかを必ず見ます。

- ローカルLLMや依存サービスが停止している  
  OpenClawのジョブだけでなく、依存しているモデルサーバーや補助サービスの起動状態も監視します。定期実行前にサービスが起動している前提を明文化します。

- 通知内容が長すぎて読まれない  
  出力は「重大な指摘」「確認すべきテスト」「補足」のように短い見出しで整理します。詳細な調査ログではなく、レビュー担当者が次に取る行動を判断できる内容にします。

## Summary

OpenClawでコードレビューを自動化するワークフローは、最小構成で開始し、ログを検証しながら段階的に拡張するのが基本です。最初は1つのレビュー補助タスクに絞り、実行時刻、対象範囲、失敗時の復旧手順を明確にします。

安定してから対象ブランチやチェック内容を広げることで、レビュー品質を保ちながら運用負荷を下げられます。定期実行は導入そのものよりも、ログ確認と改善サイクルまで含めて設計することが重要です。
