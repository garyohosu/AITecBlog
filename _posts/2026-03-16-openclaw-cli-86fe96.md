OpenClaw CLI を使った日常運用では、最初から多機能に広げるより、確認系コマンドと最小の実行設定を先に固めるほうが安定します。特に、手動実行で挙動を確認してから定期実行へ進める流れにすると、切り分けが速くなります。

結論: OpenClaw CLI リファレンス：よく使うコマンド集 は、最小構成で始めてログを見ながら段階的に広げるのが最短です。

## Background

毎日の運用タスクは、手作業のままだと実行漏れや設定ブレが起きやすくなります。CLI で実行手順を固定すると、誰が触っても同じ条件で再現しやすくなります。

重要なのは、いきなり定期実行に入ることではありません。まずは「対象ジョブを確認する」「1回だけ手動で流す」「問題がなければスケジュール化する」の順に進めることです。

## Step-by-step

1. 対象ジョブを確認する

最初に、どのジョブが登録済みかを確認します。名前の取り違えを防ぐだけでも、無駄な再実行や誤操作をかなり減らせます。

```bash
# 登録済みジョブの一覧を確認
openclaw jobs list
```

2. まずは手動で1回実行する

定期実行を組む前に、対象ジョブを明示して単発実行します。成功時の出力と失敗時のログをここで確認しておくと、後の運用が安定します。

```bash
# 対象ジョブを1回だけ手動実行
openclaw jobs run --name "daily-sample"
```

ジョブ名は `openclaw jobs list` で確認した値をそのまま使ってください。似た名前のジョブがある環境では、ここを曖昧にしないことが重要です。

3. 最小構成で Cron ジョブを登録する

手動実行で問題がなければ、最小構成で定期実行を追加します。最初は 1 ジョブ 1 目的に絞ると、障害時の影響範囲を小さくできます。

```bash
# 毎日 09:00 JST に実行する最小例
openclaw cron add \
  --name "daily-sample" \
  --cron "0 9 * * *" \
  --tz "Asia/Tokyo" \
  --session isolated \
  --message "ジョブを実行して結果を要約"
```

`--tz` は必ず明示してください。JST 運用なら `Asia/Tokyo` を固定しておくと、サーバー側のデフォルト時刻に引きずられません。

4. 登録内容を確認する

登録後は、その場で設定値を見直します。実行時刻、ジョブ名、タイムゾーンの3点だけでも確認しておくと、初回トラブルを防ぎやすくなります。

```bash
# 登録済み Cron ジョブを確認
openclaw cron list
```

## Common pitfalls

- タイムゾーンを指定せず、想定より9時間ずれて実行してしまう  
`openclaw cron add` では `--tz "Asia/Tokyo"` を明示してください。登録後に `openclaw cron list` を実行し、反映値を目視で確認します。

- ジョブ名をうろ覚えで指定し、別ジョブを実行してしまう  
手動実行前に `openclaw jobs list` を見て、正式なジョブ名をそのまま使います。運用では略称ではなく、固定の命名規則を決めておくと安全です。

- 手動実行を省いて、そのまま定期実行に入れてしまう  
最初の投入では必ず `openclaw jobs run --name "..."` で単発実行してください。失敗時のログ確認と再実行手順を先に固めておくと、本番スケジュール投入後の復旧が速くなります。

- 認証切れを見落として、ジョブ設定の不具合だと誤認する  
ジョブ追加や実行が急に失敗した場合は、まず CLI の認証状態を確認します。設定を何度も直す前に、認証情報の有効期限や実行ユーザーの権限を点検してください。

- 依存サービス停止で実行だけ通り、期待した結果が出ない  
ローカル LLM や外部 API に依存するジョブは、実行前に依存先の稼働確認を入れます。ジョブ本体より先に依存先を疑うだけで、切り分け時間を短縮できます。

## Summary

OpenClaw CLI でよく使うのは、`openclaw jobs list`、`openclaw jobs run --name "..."`、`openclaw cron add`、`openclaw cron list` の4つです。まず確認、次に手動実行、最後に定期実行という順番を守るだけで、運用の安定性は大きく上がります。

最初から複雑なジョブ構成を組まず、1つずつ追加してログで検証してください。CLI の操作手順を固定すると、日々の運用ミスと復旧コストをどちらも抑えやすくなります。
